기존 소스코드 및 시스템 구조 분석

기존 소스는 유지되지만 최신 환경과의 호환성 불명구조/의존성/외부 연계 파악 필수
주요 이슈
  • 프레임워크 버전 노후화 (예: Spring, 전자정부프레임워크 등)
  • 외부 API 변경 또는 중단 가능성
  • 숨겨진 하드코딩(경로, 계정 등)
추진방향
  • 정적 분석 + 실행 테스트 병행
  • "재사용 vs 재개발" 기준 수립 (모듈 단위)
  • 핵심 기능(회원, 인증, DB연계) 우선 검증 후 단계적 확장

인프라 및 실행환경 재구성

Dev/Test/Prod 분리

개발/테스트/운영 환경 분리 요구
주요 이슈
  • 기존 운영환경 미존재 또는 재현 불가
  • OS/미들웨어 버전 불일치
  • 라이선스 및 인증서 만료
추진방향
  • 컨테이너(Docker) 기반 환경 표준화 추천
  • IaC (Terraform 등)로 환경 재현성 확보
  • SSL, WAS, DB 버전 최신화 및 보안패치 적용

데이터 복구 및 초기 데이터셋 구성

DB 구조는 존재, 데이터는 대부분 삭제된 상태
주요 이슈
  • 실제 서비스 재개 시 데이터 부족
  • 데이터 정합성 및 참조 무결성 깨질 가능성
  • 개인정보 삭제 후 재구성 필요
추진방향
  • 잔존 데이터(덤프/파일) 전수 조사
  • 최소 운영 가능한 "Seed 데이터" 설계
  • 코드 테이블, 기준정보 먼저 복구
  • 데이터 클렌징 + 표준화 병행

개인정보 및 보안 체계 재정립

기존 개인정보는 이미 파기됨보안 요구사항 매우 강함 (암호화, 로그, 취약점 등)
주요 이슈
  • 인증/회원 기능 재설계 필요
  • 암호화 정책 미적용 가능성
  • 로그/접근통제 미흡
추진방향
  • 비밀번호: SHA-256 이상 단방향 암호화
  • 개인정보 최소 수집 + 비식별화 설계
  • OWASP Top 10 기반 취약점 점검 자동화
  • 관리자 접근 VPN/IP 제한

데이터 연계 및 외부 시스템 호환성

문화체육관광부 공공데이터 시스템 연계 필수
주요 이슈
  • 연계 표준(API/메타데이터) 변경 가능성
  • 방화벽/포트/에이전트 설치 문제
  • 데이터 구조 불일치
추진방향
  • 연계 인터페이스 별도 계층(API Gateway) 구성
  • 메타데이터 기반 연계 설계
  • 비동기 처리(Queue) 도입으로 안정성 확보

테스트 및 베이스라인(Baseline) 확보

기능/연계/보안 테스트 후 기준 상태 확보 요구
주요 이슈
  • 테스트 기준 데이터 부족
  • 기존 기능 정상 동작 여부 불명
  • 성능 병목 사전 식별 어려움
추진방향
  • 테스트 자동화 (단위/통합/회귀)
  • 더미 데이터 생성 전략 수립
  • "Baseline 버전" 확정 후 점진 개선

운영 안정성 및 장애 대응 체계

상시 유지관리, 장애 대응, 기술지원 요구
주요 이슈
  • 초기 장애 빈번 가능성
  • 운영 인력 의존도 높음
  • 사용자 민원 대응 부담
추진방향
  • 모니터링 (APM + 로그 수집) 필수
  • 장애 대응 Runbook 문서화
  • SLA 기반 운영 프로세스 수립

데이터 품질 및 표준화 체계 구축

  • 코드체계 혼재

추진방향

  • 코드/도메인 관리 체계 구축
  • 데이터 품질 진단 자동화

형상관리 및 개발 거버넌스 구축

형상관리, 접근권한, 절차관리 요구
